**HLA Laboratory System** — настольное приложение для HLA-лаборатории, которое собирает в одном рабочем решении импорт результатов, ведение файловой базы, подготовку документов, поиск по архиву и анализ динамики антител. Приложение снимает значительную часть ручной работы, снижает риск человеческой ошибки и облегчает работу врачей при ведении пациентов.

Приложение работает с типовыми файлами лабораторного процесса: CSV-выгрузками HLA-анализатора, XLSX-отчётами и заключениями в форматах JPEG, DOCX и PDF. При ручном ведении этих процессов данные и документы легко расходятся между папками, таблицами и отдельными файлами; здесь они обрабатываются как единая операция с проверками, откатом при ошибке и защитой управляемых файлов от случайного изменения.

Для лаборатории это обеспечивает несколько практических преимуществ:

- уменьшение повторного ручного ввода и копирования данных между файлами и базами;
- минимальный риск ошибки в имени пациента, дате исследования, HLA-классе, PRA и составе импортируемой пары CSV;
- единая структура архива по пациентам и датам исследований;
- быстрый поиск по файловой базе вместо ручного обхода папок;
- автоматическое создание отсортированных XLSX-отчётов и сохранение исходных CSV;
- поддержка JPEG, DOCX и PDF-заключений в одном сценарии;
- возможность восстановить управляемые файлы после повреждения или потери;
- графическая история антител для анализа пациентов;
- подготовка данных для клинического и научного анализа;
- подготовка данных для `автоматизированного` выполнения `виртуального кросс-матча`.

Приложение рассчитано на лаборатории, рабочие процессы которых совместимы с поддерживаемой моделью хранения исследований. Основной сценарий связан с трансплантологией и ведением пациентов после трансплантации органов, но сама логика может быть адаптирована под другие HLA-лаборатории со схожими рабочими процессами.

В работе используются два основных источника данных:

- **файловая база** на локальном или сетевом диске, где хранятся исходные CSV, заключения, сформированные XLSX/PDF/DOCX-документы и архив пациента;
- **PostgreSQL**, где приложение хранит структурированные сведения, необходимые для импорта, поиска, восстановления управляемых файлов, анализа динамики антител, аналитической обработки данных и выполнения `виртуального кросс-матча` пары `донор-реципиент`.

Дополнительно приложение может подключать вторую базу PostgreSQL для сценария `📈 Динамика антител`. Она используется как источник чтения для общей хронологии исследований и не заменяет основную рабочую базу приложения.

Также поддерживается опциональная внешняя картотека пациентов. Она нужна только для быстрого поиска паспортных данных и автозаполнения раздела `👤 Данные пациента`; она не хранит результаты HLA-исследований и не подменяет рабочую базу приложения.

## Назначение и функции

![Экран запуска](assets/startup.png)

Приложение объединяет несколько рабочих задач в одном интерфейсе:

- импорт и сохранение исходных CSV-файлов, выгруженных с HLA-анализатора;
- импорт и автоматическое сохранение заключений в форматах JPEG, DOCX и PDF;
- создание и сопровождение файловой базы по профилям, пациентам и датам исследований;
- создание сгруппированных по низкому разрешению и отсортированных по MFI XLSX-отчётов об уровне антител;
- синхронизацию импортированных исследований с базой PostgreSQL;
- служебное восстановление повреждённых или утраченных управляемых файлов по данным, сохранённым в PostgreSQL;
- проверку совместимости пары Class I / Class II для одного исследования;
- поиск пациентов, исследований и файлов через боковой проводник и ускорение поиска с помощью локального индекса;
- автозаполнение раздела `👤 Данные пациента` из внешней картотеки, если она настроена;
- построение интерактивной динамики антител по основной и, при наличии, дополнительной базе данных;
- постепенный перевод исторических папок пациентов к новому формату именования;
- запуск в ограниченном режиме, если файловая база или PostgreSQL временно недоступны.

## Основные возможности

### 1. Импорт результатов HLA-анализа

Приложение работает с CSV фиксированной структуры, полученными с анализатора:

- Class I Single Antigen Results;
- Class II Single Antigen Results.

Во время импорта приложение:

- считывает данные пациента, дату исследования, PRA и таблицу антител;
- определяет класс исследования;
- сверяет пару Class I / Class II по пациенту и дате партии;
- связывает исследование с выбранным профилем;
- принимает файл заключения в формате JPEG или DOCX;
- сохраняет итоговый файл заключения в PDF, если исходным заключением был DOCX-документ;
- сохраняет исходный CSV в файловую базу;
- формирует XLSX-файл для ежедневной работы;
- создаёт новую папку пациента или использует существующую;
- синхронизирует импортированное исследование с PostgreSQL;
- выполняет предварительные проверки, чтобы не перезаписать связанные данные некорректным файлом.

Импорт отрицательного скрининга, когда выбран только файл заключения без CSV, остаётся файловым сценарием. Такое исследование сохраняется в файловой базе, но не создаёт исследовательскую запись в PostgreSQL.

### 2. Разделение файловой и исследовательской модели

**Файловая база** остаётся полным рабочим архивом лабораторных документов. В ней хранятся исходные CSV, сформированные XLSX-файлы, PDF/JPEG-заключения, сопутствующие файлы пациента и исторические папки.

**PostgreSQL** хранит структурированную часть исследовательских данных: карточки пациентов, сведения об импортированных исследованиях, строки антител для положительных CSV и служебные данные, необходимые для согласованности файловой базы.

Такое разделение позволяет:

- сохранить привычную файловую организацию архива;
- строить быстрый поиск и аналитику по структурированным данным;
- восстанавливать управляемые файлы после повреждения или потери;
- согласованно удалять или заменять исследования без ручной чистки связанных записей;
- использовать PostgreSQL для межклиентской блокировки записи в сетевой работе.

### 3. Боковой проводник и поиск

В боковой панели доступен встроенный проводник. Он умеет:

- показывать файловую базу по профилям, пациентам и исследованиям;
- искать по имени файла или папки;
- использовать локальный индекс для быстрого поиска;
- переключаться на резервный поиск по дереву, если индекс временно недоступен;
- открывать найденные объекты прямо в дереве;
- сортировать папки исследований по календарной дате;
- выполнять допустимые операции с файлами и папками;
- блокировать опасные операции с управляемыми служебными файлами;
- оставаться доступным для просмотра и поиска в сборках, где отключены лабораторные действия записи.

### 4. Работа с папками пациентов

Проект поддерживает исторические и новые форматы именования папок пациентов. Новый формат включает фамилию, имя, отчество при наличии, дату рождения, пол и, если указан, код реципиента.

Приложение умеет:

- подбирать существующую папку пациента;
- создавать новую папку;
- приводить папки старого формата к новому формату;
- добавлять код реципиента к имени папки, если он указан в форме пациента;
- обновлять связанные служебные файлы и внутренние связи;
- откатывать критичные изменения при ошибке операции.

Папки нового формата без кода реципиента могут встречаться как переходный формат или использоваться в сценариях, где код реципиента неизвестен. Если код реципиента введён, новая или переименовываемая папка получает имя с этим кодом.

### 5. Формирование документов

Приложение содержит отдельные сценарии генерации документов:

- XLSX с результатами сортировки антител;
- XLSX для блока суммарного титра антител к A, B, DRB1 и DQB1;
- DOCX-заключения для клинической работы;
- PDF-копии заключений при наличии Microsoft Word.

Папки сохранения документов задаются в пользовательских настройках приложения. Если Word на рабочем месте недоступен, сценарии, требующие преобразования DOC/DOCX в PDF, будут ограничены.

### 6. 📈 Динамика антител

Окно `📈 Динамика антител` предназначено для просмотра исследовательской истории пациента по основной PostgreSQL и, при необходимости, по дополнительной базе данных.

Окно позволяет:

- открывать пациента по коду реципиента и профилю или по персональным данным;
- отображать сведения о пациенте из основной рабочей базы;
- строить синхронные графики Class I и Class II на общей временной оси;
- переключать детализацию между низким и высоким разрешением;
- ограничивать отображение отдельными группами антител;
- менять нижний порог чувствительности без повторной загрузки данных;
- использовать дополнительную БД в режиме чтения как источник более ранних или более поздних исследований;
- визуально отделять участки истории, полученные из разных источников.

По бизнес-логике основная PostgreSQL остаётся единственным обязательным источником и единственной базой, в которую приложение записывает результаты импорта. Дополнительная БД используется только для чтения. Если она недоступна или в ней нет данных пациента, окно динамики продолжает работу по основной БД.

Штатный сценарий предполагает, что основная и дополнительная базы описывают разные хронологические участки истории пациента. Если в реальной лабораторной практике эти участки пересекаются по одному и тому же классу исследования в одну дату, правила интерпретации графиков нужно согласовать до промышленного использования.

### 7. Устойчивость и защита данных

Для рабочих сценариев лаборатории важна не только функциональность, но и отказоустойчивость. В приложении предусмотрены:

- проверка доступности файловой базы и PostgreSQL при запуске;
- ограниченный режим при временной недоступности внешних ресурсов;
- запрет второго экземпляра приложения в том же профиле Windows;
- локальный индекс файлового дерева вместо постоянного обхода сетевого диска;
- межклиентская блокировка записи при работе нескольких рабочих мест с общей файловой базой и PostgreSQL;
- отключение операций записи в специальных сборках просмотра;
- снимок состояния и откат критичных файловых операций;
- защита управляемых путей от случайного удаления или некорректного переименования;
- фоновая перестройка индекса;
- предварительные проверки перед заменой или удалением импортированных исследований;
- отдельный защищённый режим восстановления управляемых файлов без изменения записей PostgreSQL.

### 8. Восстановление файловой базы

Для аварийных ситуаций, когда управляемые файлы исследования отсутствуют или повреждены в файловой базе, предусмотрен служебный режим восстановления.

Режим восстановления:

- запускается отдельно от обычного рабочего сценария;
- читает сохранённые в PostgreSQL сведения об импортированных исследованиях;
- проверяет целостность исходных данных;
- восстанавливает управляемые CSV и XLSX-файлы;
- при необходимости заново формирует PDF-заключение;
- не перезаписывает уже корректные файлы;
- использует ту же межклиентскую блокировку записи, что импорт и защищённые операции бокового проводника;
- формирует текстовый отчёт о выполнении.

Восстановление не изменяет PostgreSQL и не заменяет резервное копирование файловой базы. Отрицательные скрининги, импортированные только как файл заключения без CSV, а также произвольные сопутствующие файлы пациента не могут быть восстановлены из исследовательских данных PostgreSQL.

## Типовой рабочий сценарий

1. Лаборант получает один или два CSV-файла с анализатора.
2. В приложении выбирается профиль и выполняется импорт результатов.
3. Приложение находит существующую папку пациента или предлагает создать новую.
4. Исходные CSV, XLSX и заключение сохраняются в файловую базу.
5. Структурированные данные исследования синхронизируются с PostgreSQL.
6. Врач открывает исследование через боковой проводник, поиск или карточку пациента.
7. При необходимости формируются дополнительные документы.
8. Для ретроспективного анализа открывается окно динамики антител.

## Состав приложения

Пользовательская часть приложения включает:

- главное окно с формой пациента, выбором профиля, импортом и документами;
- боковой проводник файловой базы;
- окно динамики антител;
- диалоги настроек, подтверждений и выбора совпадений;
- фоновые операции для импорта, восстановления, поиска и обновления индекса;
- механизмы хранения пользовательских настроек и паролей в системном хранилище Windows;
- эксплуатационные сценарии для задач администратора.

Техническая структура исходного кода, имена внутренних модулей, точные команды сборки и низкоуровневые схемы данных не публикуются в пользовательской документации дистрибутива. Эти сведения относятся к внутренней документации сопровождения и предоставляются только в разрешённом объёме.

Такое ограничение публикации связано с повышением безопасности рабочих установок. Приложение предназначено для процессов, где обрабатываются реальные медицинские и персональные данные пациентов.

В публичном описании намеренно не приводятся:

- точная структура пакетов и внутренних модулей;
- команды запуска, сборки и параметры упаковки;
- DDL-схемы, индексы, ограничения и служебные таблицы;
- реальные сетевые адреса, имена БД, пути рабочих ресурсов и параметры инфраструктуры;
- сведения, которые помогают угадывать расположение, состав или правила доступа к рабочим данным пациентов;
- подробные алгоритмы разбора CSV, сопоставления пациентов и восстановления файлов;
- внутренние инструкции администратора и регламент развёртывания.

## Технологии и инструменты

В проекте используются технологии и подходы, характерные для прикладных desktop-систем на Python:

- **Python** как основной язык приложения и сервисной логики;
- **Qt/PySide6** для настольного интерфейса, диалогов, фоновых задач и desktop UX;
- **PostgreSQL** и **SQLAlchemy** для структурированных исследовательских данных, проверок доступности и межклиентской координации;
- **SQLite** для локального индекса файловой базы и ускорения поиска;
- **pandas** и **openpyxl** для обработки табличных данных и формирования XLSX-отчётов;
- **python-docx** для генерации DOCX-документов;
- **pyqtgraph** для интерактивной визуализации динамики антител;
- **pywin32/COM** для интеграции с Microsoft Word при преобразовании документов;
- **keyring** и Windows Credential Manager для хранения пользовательских секретов вне текстовых конфигураций;
- **ODBC/DB-драйверы** для подключения к внешним справочникам пациентов при необходимости;
- **PowerShell** и Планировщик заданий Windows для утверждённых административных сценариев обслуживания;
- **Nuitka** для подготовки автономной Windows-поставки без требования устанавливать Python на рабочем ПК.

На рабочем ПК для запуска собранного приложения отдельная установка Python не требуется. Если используется преобразование DOCX-документов в PDF, на этом рабочем месте должен быть установлен Microsoft Word.

## Системные требования

Рекомендуемая среда:

- Windows 10 или Windows 11;
- доступ к локальной или сетевой файловой базе;
- доступ к PostgreSQL для полнофункционального режима импорта и анализа;
- Microsoft Word, если требуется импорт DOC/DOCX-заключений с автоматическим сохранением в PDF;
- драйвер или клиент внешней БД, если используется опциональная картотека пациентов;
- права пользователя на чтение и запись в рабочую файловую базу;
- права на запуск приложения и сохранение пользовательских настроек в профиле Windows.

Если в работе используются только JPEG-заключения и печать DOCX из приложения не требуется, Microsoft Word для основных сценариев не нужен.

### Важно для преобразования DOCX в PDF

Если используется импорт заключения из DOCX с автоматическим сохранением в PDF, одной из частых причин ошибок являются ограничения безопасности Microsoft Word при открытии документа для преобразования.

Чтобы Word корректно выполнял преобразование, рабочую папку с документами следует добавить в надёжные расположения Word. Если используется сетевой ресурс, необходимо также разрешить надёжные расположения в сети. В организациях с групповыми политиками эти параметры должен подтвердить администратор.

## Адаптация под другую лабораторию

Приложение может быть адаптировано под разные лабораторные конфигурации. Обычно адаптация включает:

Перед вводом адаптированной конфигурации в работу необходимо проверить:

4. переименование исторических папок;

## Файловая и реляционная модель данных

### Файловая база

Файловая база организована по рабочим профилям. Внутри профиля находятся папки пациентов, а внутри папок пациентов — исследования по датам. В папке исследования хранятся управляемые файлы, созданные или принятые приложением: исходные CSV, XLSX-файлы и файлы заключений.

Приложение сохраняет совместимость с историческими папками пациентов, но при новых операциях постепенно переводит структуру к более полному формату имени. Это помогает различать однофамильцев, учитывать дату рождения, пол и код реципиента.

В файловой базе могут присутствовать служебные области, которые используются приложением для согласованного хранения исходных импортированных данных. Их не следует редактировать вручную.

### PostgreSQL

PostgreSQL используется для структурированных исследовательских данных:

- идентификация пациента в рамках выбранного профиля;
- сведения об импортированных исследованиях;
- результаты разбора антител;
- данные для построения динамики;
- служебные связи между файловой базой и исследовательскими записями;
- данные для восстановления управляемых файлов.

Полным архивом лабораторных документов остаётся файловая база. PostgreSQL не заменяет хранение всех сопутствующих файлов пациента и не является единственным источником восстановления произвольных документов.

Точная схема PostgreSQL, индексы и служебные ограничения относятся к внутренней документации развёртывания. На рабочем месте база должна создаваться ответственным администратором по утверждённой инструкции для конкретной площадки.

### Дополнительная PostgreSQL

Дополнительная PostgreSQL используется только для чтения в окне динамики антител. Она позволяет объединять историю пациента из двух хронологических участков. Импорт новых исследований, создание карточек и обычная работа с файловой базой выполняются через основную PostgreSQL.

При недоступности дополнительной БД приложение продолжает работу с основной БД. Если дополнительный источник используется в рабочем процессе, его параметры должны быть согласованы для всех рабочих мест, подключённых к одной файловой базе.

### Опциональная картотека пациентов

Картотека пациентов — это внешний источник паспортных данных только для чтения. Она применяется в форме пациента для подсказок, поиска по коду реципиента и автозаполнения полей.

Картотека:

- не создаёт и не переименовывает папки пациентов;
- не сохраняет результаты исследований;
- не записывает данные в PostgreSQL;
- не участвует в откатах импорта;
- не подменяет основную базу приложения;
- может использовать локальный снимок для работы при временной недоступности внешнего источника.

Если внешняя картотека недоступна, приложение продолжает работать по локальной файловой базе и основной PostgreSQL. Для рабочего развёртывания доступ к картотеке должен выдаваться по принципу минимальных прав.

## Настройки

Основные параметры задаются в окне `Настройки`:

- путь к файловой базе;
- параметры подключения к основной PostgreSQL;
- папки сохранения заключений и суммарного титра;
- ограничения бокового проводника;
- параметры лабораторной конфигурации, доступные пользователю;
- парольные параметры, которые сохраняются в системном хранилище Windows.

Параметры дополнительной PostgreSQL открываются из окна `📈 Динамика антител` и сохраняются отдельно. Если дополнительная БД использует те же учётные данные и тот же адрес подключения, приложение может использовать пароль основной PostgreSQL.

Пароли основной PostgreSQL, дополнительной PostgreSQL и внешней картотеки не следует хранить в текстовых конфигурационных файлах. Для рабочей установки они задаются через штатный режим настройки паролей установленного приложения и сохраняются в системном хранилище учётных данных Windows для конкретного пользователя.

Коды подтверждения для защищённых действий должны передаваться только ответственным пользователям. Их назначение — снизить риск случайного импорта, удаления, изменения настроек или запуска административного сценария.

## Режимы эксплуатации

Приложение поддерживает две рабочие схемы:

- **однопользовательская**: приложение, файловая база и PostgreSQL работают на одном рабочем месте;
- **многопользовательская**: несколько рабочих мест используют одну общую файловую базу и одну согласованную PostgreSQL.

В одном профиле Windows одновременно может работать только один экземпляр приложения. Это защищает локальные пользовательские настройки, временные каталоги и индекс бокового проводника от конфликтов между двумя запусками на одном рабочем месте.

### Однопользовательский режим

Этот вариант подходит, если файловая база и PostgreSQL находятся на том же ПК, где запускается приложение.

Перед вводом в работу нужно:

1. создать рабочую папку файловой базы;
2. подготовить PostgreSQL;
3. задать параметры подключения в настройках приложения;
4. настроить пароли через штатный режим установленного приложения;
5. выполнить тестовый импорт;
6. проверить формирование документов;
7. проверить поиск по файловой базе;
8. проверить восстановление на тестовом наборе, если этот сценарий будет использоваться.

### Многопользовательский режим

В многопользовательском режиме одно рабочее место или сервер хранит рабочую файловую базу и PostgreSQL, а остальные клиенты подключаются к ним по сети.

Правила такого режима:

- все рабочие места должны использовать одну и ту же версию приложения;
- все клиенты должны работать с одной рабочей файловой базой, а не с её зеркальной копией;
- основная PostgreSQL должна быть общей для всех клиентов этой файловой базы;
- параметры дополнительной PostgreSQL, если она используется, должны быть согласованы на всех клиентах;
- одновременно записывать данные может только один клиент;
- при недоступности PostgreSQL сетевой режим записи недоступен;
- служебные сценарии зеркалирования не заменяют рабочую файловую базу для клиентов.

Настройка общего доступа, сетевых разрешений, брандмауэра и PostgreSQL выполняется администратором площадки по внутренней инструкции. В пользовательском README эти параметры намеренно не приводятся, потому что они зависят от инфраструктуры организации.

### Служебное восстановление файловой базы

Служебное восстановление предназначено для ответственного восстановления управляемых файлов после потери или повреждения части файловой базы. Оно использует текущие настройки пользователя Windows и доступные источники PostgreSQL.

Перед запуском восстановления должны выполняться условия:

- рабочая файловая база доступна;
- обязательная PostgreSQL доступна;
- папки заключений настроены корректно;
- при необходимости формирования PDF доступен Microsoft Word;
- у пользователя есть код подтверждения административного действия;
- другие клиенты не выполняют импорт, удаление или изменение той же файловой базы.

По завершении создаётся отчёт, где указано, сколько исследований было прочитано, обработано, восстановлено или пропущено, а также причины пропусков.

### Минимальная приёмка перед вводом в эксплуатацию

Перед рабочим использованием рекомендуется проверить:

1. успешный импорт на основном рабочем месте;
2. успешный импорт на одном дополнительном клиенте, если используется сеть;
3. корректную блокировку параллельной записи;
4. поиск, боковой проводник и открытие исследований;
5. формирование XLSX и DOCX/PDF;
6. работу окна `📈 Динамика антител`;
7. работу внешней картотеки, если она настроена;
8. отсутствие несогласованных записей после тестовых импортов и удалений;
9. поведение приложения при временной недоступности файловой базы;
10. поведение приложения при временной недоступности PostgreSQL.
